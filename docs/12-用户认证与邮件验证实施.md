# FutureMoney 用户认证与邮件验证实施

> 规划日期：2026-07-23。本文定义阶段 B 的实现边界：在现有 `future-money.pages.dev` 上增加安全、可恢复的邮箱账号体系，但暂不上传和同步用户资金数据。

## 一、阶段目标

阶段 B 完成后，用户可以：

1. 使用邮箱和密码注册；
2. 接收并完成邮箱验证；
3. 登录、刷新页面后保持会话、退出；
4. 发起忘记密码并通过邮件重置；
5. 在顶栏查看当前登录用户；
6. 未登录时继续作为游客使用本地功能。

本阶段只建立“用户是谁”和“会话是否有效”，资金数据仍保存在当前浏览器。云端资金快照、游客数据迁移和多设备冲突属于阶段 C、D。

## 二、技术方案

- **认证框架**：Better Auth；
- **数据库适配**：Drizzle + Cloudflare D1；
- **运行位置**：Cloudflare Pages Functions；
- **API 前缀**：`/api/auth/*`；
- **会话方式**：`HttpOnly + Secure + SameSite=Lax` Cookie；
- **防刷**：Cloudflare Turnstile；
- **邮件**：`AuthEmailSender` 抽象，首个生产实现使用 Resend HTTP API；
- **前端**：登录/注册抽屉、邮箱验证结果页、重置密码页、顶栏用户菜单。

不自行实现密码哈希、Session Token、重置令牌和邮箱验证令牌。

## 三、部署关系

```text
https://future-money.pages.dev
  │
  ├─ /assets/*                  React 静态资源
  ├─ /api/v1/health             Hono 工程健康检查
  └─ /api/auth/*                Better Auth
       │
       ├─ Turnstile 校验
       ├─ D1 用户/会话/令牌表
       └─ AuthEmailSender -> Resend
```

前端和认证 API 同源，不增加 CORS 白名单和第二个认证域名。

## 五、认证数据表

Better Auth 默认需要以下表：

- `user`：用户 ID、姓名、邮箱、验证状态、创建时间；
- `session`：会话令牌、过期时间、IP、User-Agent；
- `account`：密码凭据和未来 OAuth 账号关联；
- `verification`：邮箱验证和密码重置令牌。

表结构通过 Better Auth CLI/官方 schema 与 Drizzle migration 生成，不手工猜测字段。阶段 B 不创建 `user_vaults`。

## 六、接口与页面流程

### 6.1 注册

1. 用户输入邮箱、密码和确认密码；
2. 前端获取 Turnstile token；
3. token 通过 `x-turnstile-token` 请求头提交；
4. Pages Function 先校验 Turnstile；
5. Better Auth 创建未验证用户；
6. 发送验证邮件；
7. 界面进入“请检查邮箱”，不泄露账号是否已存在的额外信息。

### 6.2 邮箱验证

验证链接回到同一站点，由 Better Auth 校验一次性令牌。成功后刷新 Session，失败时提供重新发送入口。

### 6.3 登录

- 未验证邮箱不能开启云同步；
- 登录失败使用统一错误提示，不区分“邮箱不存在”和“密码错误”；
- 成功后关闭抽屉并刷新顶栏 Session；
- 游客本地数据暂不自动上传，阶段 C 再弹出迁移选择。

### 6.4 忘记密码

无论邮箱是否存在，界面都显示相同的“如果账号存在，邮件已发送”。重置链接为一次性、短有效期令牌。

### 6.5 退出

退出只销毁服务端 Session，不删除当前浏览器游客数据，也不在阶段 B 切换 localStorage key。

## 七、密码与会话要求

- 密码最少 10 位，最多 128 位；
- 前端提供强度说明，但服务端始终再次校验；
- 不允许把密码、Session Token、Cookie 或邮件令牌写入日志；
- Cookie 在生产环境必须启用 `Secure`；
- `trustedOrigins` 只包含正式 Pages 域名、本地 Pages 地址和明确的预览域名策略；
- 所有认证响应禁用敏感缓存；
- 登录和注册表单不保存到项目 localStorage。

## 八、Turnstile

保护以下入口：

- 注册；
- 邮箱密码登录；
- 忘记密码；
- 重新发送验证邮件。

前端公开配置：

```text
VITE_TURNSTILE_SITE_KEY
```

服务端 Secret：

```text
TURNSTILE_SECRET_KEY
```

生产环境缺少 Secret 时必须拒绝受保护请求；本地测试通过依赖注入使用 mock，不在代码里写固定绕过值。

## 九、事务邮件

服务端统一接口：

```ts
interface AuthEmailSender {
  sendVerificationEmail(input: { email: string; url: string }): Promise<void>;
  sendPasswordResetEmail(input: { email: string; url: string }): Promise<void>;
}
```

生产配置：

```text
RESEND_API_KEY
AUTH_FROM_EMAIL
```

邮件正文只包含产品名称、操作说明、有效期提醒和一次性链接，不包含资金数据。发送失败应记录不含邮箱与链接的错误类型，并允许用户稍后重试。

## 十、环境变量与 Secret

```text
BETTER_AUTH_URL=https://future-money.pages.dev
BETTER_AUTH_SECRET=<至少 32 字节随机值>
TURNSTILE_SECRET_KEY=<Cloudflare Secret>
RESEND_API_KEY=<Resend Secret>
AUTH_FROM_EMAIL=FutureMoney <no-reply@已验证域名>
```

只有 `VITE_TURNSTILE_SITE_KEY` 可以进入前端构建。其余全部配置为 Pages Functions Secret，不进入 `.env`、GitHub、文档或聊天记录。

## 十一、界面原则

- 顶栏保持当前资金工作区主结构，不增加后台式导航；
- 未登录显示轻量“登录”入口；
- 登录、注册、忘记密码使用右侧抽屉，保持现有产品交互语言；
- 登录后顶栏显示头像/邮箱缩写和用户菜单；
- 明确区分“本地数据”和“账号已登录”，避免用户误以为已经云端保存；
- 阶段 C 完成前，登录成功后仍显示“资金数据仅保存在本机”。

- 表单校验与提交状态；

- Session 加载、已登录用户菜单和退出；
- 登录成功后不会自动覆盖或上传游客数据。

## 十三、实施顺序

### B1：后端认证骨架

- 安装 Better Auth；
- 生成 Drizzle 认证 schema 和 migration；
- 挂载 `/api/auth/*`；
- 完成 Session API 与单元测试。

### B2：前端认证界面

- Better Auth client；
- 登录/注册/忘记密码抽屉；
- 顶栏 Session 和用户菜单；
- 验证/重置结果页面。

### B3：Turnstile 与邮件

- Turnstile 前后端；
- Resend adapter；
- 邮件模板；
- Secret 配置和生产故障处理。

### B4：生产验收

- 远程 D1 migration；
- 注册、验证、登录、刷新、退出、重置全流程；
- Pages 预览和正式地址验收；
- 日志与 D1 数据安全检查。

## 十四、用户介入节点

开发 B1、B2 不需要用户准备外部账号。进入 B3 时需要：

1. 确定用于发信的域名或子域名；
2. 登录 Resend 并完成 DNS 验证；
3. 在 Cloudflare Turnstile 创建正式站点；
4. 浏览器中确认必要授权。

Secret 由命令行安全写入 Cloudflare，用户不需要把值发到聊天中。

## 十五、完成标准

- 全部认证表 migration 成功；
- 本地和 Pages 环境认证 API 正常；
- 注册到邮箱验证、登录、退出、密码重置全流程通过；
- CI、Pages 自动部署和现有本地功能不受影响；
- 未登录用户仍可完整使用本地模式；
- 登录不会静默上传、覆盖或删除任何财务数据。
