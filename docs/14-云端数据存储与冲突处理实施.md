# FutureMoney 云端数据存储与冲突处理实施

> 实施日期：2026-07-23。对应《用户注册与 Cloudflare 云端存储规划》的阶段 C、D。

## 一、完成范围

- 新增 `user_vaults`，每名用户只保存一份完整资金快照；
- Worker 使用 AES-256-GCM 加密后写入 D1，数据库中不保存明文账户、金额或备注；
- 所有 vault 和账号删除接口先验证 Better Auth Session，`user_id` 只来自服务端会话；
- localStorage 从单一 `future-money` 迁移为游客/用户隔离键；
- 登录后显示“检查、仅本机、同步中、已同步、离线、冲突、错误”等持久状态；
- 首次登录必须选择上传本机、使用云端或暂时仅存本机；
- 使用单调递增 `revision` 做乐观锁，旧版本保存返回 `409 Conflict`；
- 提供分别导出本机/云端版本、删除云端副本和注销账号入口。

## 二、存储结构

```text
localStorage
├─ future-money:guest
├─ future-money:user:{userId}
└─ future-money:sync:{userId}  # revision、SHA-256 摘要和同步开关

D1
└─ user_vaults
   ├─ user_id
   ├─ revision
   ├─ schema_version
   ├─ key_version
   ├─ iv
   ├─ ciphertext
   └─ updated_at
```

旧版 `future-money` 首次加载时只复制到 `future-money:guest`，不立即删除旧键，避免迁移异常造成数据损失。

## 三、加密边界

生产环境必须配置 Cloudflare Pages Secret：

```text
DATA_ENCRYPTION_KEY_V1
```

它是 32 个随机字节的 Base64 编码，只能通过 Cloudflare Secret 写入，不得进入 Git、前端 bundle、文档示例值或日志。

加密使用：

- 算法：AES-GCM；
- IV：每次保存重新生成 12 个随机字节；
- Additional Authenticated Data：当前 Better Auth `user.id`；
- `key_version=1`，为后续密钥轮换保留版本字段。

这属于服务端静态加密，并非零知识加密。拥有 Cloudflare 运行权限和主密钥的管理员理论上仍可解密。

## 四、API

```text
GET    /api/v1/vault
PUT    /api/v1/vault
DELETE /api/v1/vault
DELETE /api/v1/account
```

`PUT /api/v1/vault` 接收：

```json
{
  "expectedRevision": 3,
  "data": {
    "version": 1,
    "accounts": [],
    "transactions": [],
    "series": [],
    "categories": []
  }
}
```

只有当前云端 revision 等于 `expectedRevision` 才能更新。插入和更新还会在 SQL 写入时再次检查，避免两个并发请求同时通过前置读取。

## 五、首次登录决策

| 本机 | 云端 | 行为 |
| --- | --- | --- |
| 无 | 无 | 创建空用户空间并保存第一版 |
| 无 | 有 | 自动恢复云端到该用户的本地空间 |
| 有 | 无 | 展示选择框：上传本机或暂时仅存本机 |
| 有 | 有 | 展示双方摘要，禁止自动覆盖 |

两边都有数据时可以：

1. 分别导出两份 JSON 备份；
2. 使用云端数据；
3. 明确用本机数据覆盖当前云端 revision；
4. 暂时仅保存在本机，停止自动同步。

## 六、正常同步与冲突

所有编辑先写当前用户的 localStorage，1.5 秒防抖后上传快照。同步元数据只保存 SHA-256 摘要，不复制整份账本，避免占满 localStorage。

启动时比较本机、云端和上次同步摘要：

- 两边相同：更新 revision，显示已同步；
- 仅本机变化：按最新已知 revision 上传；
- 仅云端变化：安全恢复云端；
- 两边都变化：进入冲突框；
- 保存期间又发生编辑：串行保存最新快照，避免同一设备制造假冲突；
- 断网或服务异常：本机数据继续有效，不清空、不降级覆盖。

第一期不做字段级自动合并。财务数据的“看似可合并”可能包含删除、周期组变更和同 ID 修改，自动合并的风险高于明确选择版本。

## 七、数据权利

- 删除云端资金数据：只删 `user_vaults`，本机账本保持不变并切换为“仅本机”；
- 注销账号：要求显式确认，删除 `user`，由外键级联删除 Session、Provider Account 和 vault；
- 冲突时可先导出双方版本；
- 所有数据响应使用 `Cache-Control: no-store`。

## 八、测试与验收

自动化测试覆盖：

- 未登录访问 vault 返回 `401`；
- 写入 D1 的内容不包含明文资金备注；
- 同一用户可以正确解密，换用户无法解密；
- 缺失或错误长度密钥时服务关闭而不是明文降级；
- 旧 revision 返回 `409` 且不覆盖新版本；
- 首次登录不会自动上传；
- 双边修改必定进入冲突选择；
- 注销账号必须显式确认。

本地验收命令：

```bash
npm run db:migrate:local
npm run type-check
npm test
npm run build
npm run build:functions
```

## 九、生产发布步骤

1. 生成 32 字节随机主密钥，不在终端输出；
2. 通过标准输入写入 Pages Secret `DATA_ENCRYPTION_KEY_V1`；
3. 执行远程 migration `0002_wise_ikaris.sql`；
4. 推送 `main`，继续使用现有 GitHub → Cloudflare Pages workflow；
5. 验证 `/api/v1/health`、首次登录上传、刷新恢复、第二设备冲突和删除云端副本；
6. 确认 D1 `ciphertext` 中没有明文账户名、金额备注。

生产密钥、OAuth Client Secret 和任何访问 Token 都不得写入提交或文档。

## 十一、冲突弹窗与 Modal 规范

数据选择弹窗最多包含四个操作：

1. 分别导出备份；
2. 暂时仅保存在本机；
3. 使用云端数据；
4. 用本机数据覆盖云端。

不能把四个按钮硬塞在同一行，也不能把操作栏放进 `.semi-modal-body`。最终结构为：

```text
Modal body
├─ 风险提示
└─ 本机 / 云端摘要

Modal footer
├─ 第一行：导出备份 + 说明
└─ 第二行：仅本机 / 使用云端 / 覆盖云端
```

统一约束：

- 自定义 Modal 操作按钮必须通过 `footer` 属性渲染；
- footer 固定保留 `20px` 底部安全间距；
- 决策按钮容器允许换行，不能产生水平溢出；
- `480px` 以下改为纵向按钮；
- 新 Modal 必须增加结构测试，确保按钮位于 `.semi-modal-footer` 而非 `.semi-modal-body`；
- 冲突场景测试必须覆盖三个决策按钮同时出现。

对应修复：

- `01f4331`：统一数据弹窗 footer 与底部间距；
- `1bfd1e2`：将四按钮冲突 footer 拆为两层响应式布局。
