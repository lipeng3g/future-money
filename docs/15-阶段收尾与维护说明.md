## 一、收尾结论

FutureMoney 已完成从纯本地资金推演工具到“本地优先 + 可选加密云同步”的完整主链路。本阶段不再继续扩展功能，保留现有 Cloudflare Pages 生产站点和 GitHub 自动部署。

当前已经具备：

- 多账户、分类、单笔与周期收支；
- 周期记录逐笔预览、真实生成、单笔编辑与整组续期；
- 历史 / 未来资金曲线、自定义时间范围和日周月粒度；
- 月度账本、筛选、批量操作、JSON 导入导出；
- 响应式产品布局、可收缩账户栏、现代化抽屉和首次使用说明；
- GitHub OAuth 登录；
- 本机 / 用户数据隔离；
- AES-256-GCM 加密云端快照；
- revision 乐观锁、离线状态和人工冲突选择；
- 删除云端数据、注销账号和冲突双方备份。

## 二、生产架构基线

```text
GitHub main
   │
   └─ Cloudflare Pages Git 集成
        ├─ Vite / React 静态前端
        ├─ Pages Functions / Hono API
        ├─ Better Auth / GitHub OAuth
        └─ Cloudflare D1
             ├─ user / session / account
             └─ user_vaults（加密资金快照）
```

核心组件：

| 范围 | 实现 |
| --- | --- |
| 前端 | React 19、TypeScript、Vite、Semi Design、VChart |
| 本地状态 | Zustand persist + localStorage |
| API | Cloudflare Pages Functions + Hono |
| 身份 | Better Auth + GitHub OAuth |
| 数据库 | Cloudflare D1（APAC） |
| 云端账本 | 单用户单行加密快照 |
| 部署 | GitHub `main` 自动触发 Cloudflare Pages |

不需要 VPS、Docker、常驻 Worker、MySQL、Redis 或额外付费服务器。

## 三、数据保存语义

### 3.1 游客

- 使用 `future-money:guest`；
- 数据只在当前浏览器；
- 清理站点数据会删除账本；
- 应定期导出 JSON 备份。

### 3.2 登录用户

- 使用 `future-money:user:{userId}`；
- 本机数据始终先保存，不依赖网络；
- 用户可以启用或关闭云同步；
- 云同步只保存当前完整快照，不拆成业务关系表；
- 同步元数据使用 `future-money:sync:{userId}`，只保存 revision、开关和 SHA-256 摘要。

### 3.3 首次登录

首次登录不会自动上传。根据本机和云端是否存在数据，显示：

- 上传本机数据；
- 使用云端数据；
- 暂时仅保存在本机；
- 先分别导出备份。

如果两边都有数据，必须由用户选择，不自动合并。

## 四、冲突处理原则

服务端只接受 `expectedRevision` 与云端当前 revision 相等的更新。旧设备或并发请求保存时返回 `409 Conflict`。

冲突时：

- 本机版本不会删除；
- 云端版本不会自动覆盖；
- 可以分别导出两份 JSON；
- 可以选择云端；
- 可以明确用本机覆盖云端；
- 可以暂停同步，继续仅使用本机。

第一期不实现逐字段合并。账户删除、周期组续期、批量编辑和相同 ID 修改很难安全自动合并，财务数据应优先保证“不静默丢失”。

## 五、安全状态

### 5.1 Git 仓库

已检查当前文件和完整可访问 Git 历史，没有发现真实 GitHub Token、OAuth Client Secret、Cloudflare API Token、私钥或数据加密主密钥。

以下内容必须继续保持在 Git 之外：

```text
.env
.env.*
.dev.vars
.wrangler/
```

仓库只允许出现 Secret 名称和占位说明，不允许出现真实值。

### 5.2 Cloudflare Secret

生产环境依赖以下绑定名称：

```text
BETTER_AUTH_SECRET
GITHUB_CLIENT_ID
GITHUB_CLIENT_SECRET
DATA_ENCRYPTION_KEY_V1
```

这里只记录名称，不记录值。检查时使用 `wrangler pages secret list`，它只返回名称和加密状态。

### 5.3 OAuth Secret 事件

首次 GitHub Client Secret 曾出现在浏览器自动化快照中。它从未提交到 Git，但仍按泄露处理：

1. 从 Cloudflare Pages 移除旧 Secret；
2. 暂停 GitHub Provider；
3. 在 GitHub 生成全新 Secret；
4. 安全写入 Cloudflare；
5. 重新触发 Pages Git 部署；
6. 完成真实 OAuth 登录与回调测试；
7. 从 GitHub 删除旧 Secret；
8. 清空剪贴板。

当前生产环境使用轮换后的 Secret，旧 Secret 已删除。

### 5.4 加密边界

- D1 保存 `iv + ciphertext + key_version`；
- 算法为 AES-256-GCM；
- 用户 ID 作为 Additional Authenticated Data；
- 每次保存使用随机 IV；
- 数据响应使用 `Cache-Control: no-store`。

这不是零知识加密。拥有 Cloudflare 项目权限和主密钥的管理员理论上仍可解密，因此必须保护 Cloudflare 账号和部署权限。

## 六、界面与交互约束

后续如果恢复开发，必须保持以下已验证规则：

- Modal 自定义按钮必须放入正式 `footer`；
- footer 底部保留至少 `20px` 安全间距；
- 四按钮冲突场景使用两层布局，不能挤成一行；
- 移动端操作按钮纵向排列；
- 周期生成和续期必须先展示逐笔预览；
- “每 1 个月”表示连续月份，不使用“间隔 1 月”这类歧义表述；
- 财务数据上传、覆盖、删除必须明确确认；
- 登录成功不等于已经上传账本，页面必须持续显示同步状态。

## 七、已知限制

- 目前只开放 GitHub 登录，Google Provider 尚未配置；
- 不支持邮箱密码登录；
- 不支持多币种、多语言、共享账本或团队协作；
- 不支持字段级自动冲突合并；
- localStorage 仍受浏览器容量限制；
- 服务端不对加密账本做统计或搜索；
- 没有零知识恢复码，Cloudflare 密钥丢失会导致已有密文无法解密；
- Cloudflare 免费额度不是永久承诺，应关注官方策略变化；
- 前端 VChart 包体较大，构建仍会提示 chunk size warning，但不影响当前运行；
- 没有设置独立自定义域名，继续使用 `future-money.pages.dev`。

## 八、日常使用建议

即使启用云同步，也建议：

1. 发生大批量修改前导出 JSON；
2. 每月或重要节点导出一次备份；
3. 冲突时先导出双方版本再选择；
4. 不要在公共设备长期保持登录；
5. 删除浏览器数据前确认云端状态为“已同步”。

## 九、最低维护检查

不开发期间无需运行本地程序。只有出现线上异常时执行：

```bash
curl https://future-money.pages.dev/api/v1/health
curl https://future-money.pages.dev/api/v1/auth/providers
npx wrangler pages deployment list --project-name future-money
npx wrangler pages secret list --project-name future-money
```

预期：

- health 为 `status: ok`；
- providers 为 `github: true`；
- 最新 Production Deployment 为 `Active`；
- Secret 名称完整，但不读取或输出 Secret 值。

不要为了“检查”而导出 D1 中的 ciphertext 或用户财务数据。

## 十、恢复开发清单

以后恢复开发时按顺序执行：

1. `git pull --ff-only origin main`；
2. `npm install`；
3. `npx wrangler whoami` 确认 Cloudflare 账号；
4. 检查 Pages Secret 名称，不查看真实值；
5. `npm run db:migrate:local`；
6. `npm run type-check`；
7. `npm test`；
8. `npm run build`；
9. `npm run build:functions`；
10. 本地 Pages 环境验证登录、首次迁移、保存和冲突；
11. 数据库变更先生成 migration，再本地验证，最后执行 remote migration；
12. 推送 `main` 后检查 Pages Deployment 和生产域名传播；
13. 生产验收至少覆盖 health、Provider、登录、同步状态和未登录 vault `401`。

如果改动认证、密钥、数据迁移或冲突逻辑，不能只依赖单元测试，必须做一次真实生产链路验收。
