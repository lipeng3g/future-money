# FutureMoney 社交账号授权登录调整

> 决策日期：2026-07-23。本文替代 `12-用户认证与邮件验证实施.md` 中尚未开放的邮箱注册、邮箱验证和密码找回方案。已经完成的 Better Auth、D1 用户表和 Session 基础设施继续复用。

## 一、调整原因

FutureMoney 当前没有可用于事务邮件的自有域名。Cloudflare 和 Supabase 的免费邮件能力都只能用于预先验证的收件地址或开发测试，无法为任意用户提供可靠的生产验证邮件。

因此不再继续邮箱密码注册，改用社交账号 OAuth：

- 不购买服务器；
- 不依赖事务邮件；
- 不保存用户密码；
- 保留 Cloudflare Pages、Pages Functions、D1 和 Better Auth；
- 保留游客本地模式，登录仍不会自动上传或覆盖资金数据。

## 二、首批登录方式

首批实现两个配置独立、按需开放的 Provider：

1. **GitHub**：配置简单，优先完成生产闭环；
2. **Google**：面向更广泛用户，待 OAuth Consent Screen 配置后开放。

页面不展示尚未配置的 Provider。若两个 Provider 都未配置，登录抽屉只显示“授权登录准备中”，不提供无效按钮。

暂不接入微信、QQ、手机号或用户名密码。它们分别涉及平台审核、短信成本或缺少安全恢复路径。

## 三、登录流程

```text
用户点击 GitHub / Google
  -> Better Auth /api/auth/sign-in/social
  -> Provider 授权页
  -> /api/auth/callback/{provider}
  -> Better Auth 创建或关联 user/account
  -> HttpOnly Session Cookie
  -> 返回 future-money.pages.dev
```

OAuth 回调地址固定为：

```text
https://future-money.pages.dev/api/auth/callback/github
https://future-money.pages.dev/api/auth/callback/google
```

Preview 部署不注册独立回调地址，生产授权只使用正式域名。

## 四、服务端配置

Pages Functions Secret：

```text
GITHUB_CLIENT_SECRET
GOOGLE_CLIENT_SECRET
```

Pages Functions 普通环境变量：

```text
GITHUB_CLIENT_ID
GOOGLE_CLIENT_ID
```

Secret 不进入 Git、文档、聊天或前端构建产物。

新增公开能力接口：

```text
GET /api/v1/auth/providers
```

只返回已配置 Provider 的布尔状态，不返回 Client ID、Secret 或其他认证配置。前端以该接口为唯一展示依据，避免构建变量与 Functions 配置不同步。

## 五、代码调整

删除：

- 邮箱、密码、确认密码表单；
- 注册、邮箱验证、找回和重置密码页面；
- Resend adapter 和邮件模板；
- 认证场景 Turnstile；
- 邮箱认证开关和相关测试。

保留：

- Better Auth Session；
- `user / session / account / verification` 表；
- 顶栏登录入口、用户菜单和退出；
- “资金数据仍保存在当前设备”的明确提示。

`verification` 表继续保留在 Better Auth schema 中，避免手工偏离官方表结构，也为未来其他插件预留兼容性。

## 六、账号与数据边界

- OAuth 返回的邮箱只作为 Provider 账号资料，不发送 FutureMoney 邮件；
- 不在客户端保存 OAuth Access Token；
- Better Auth 负责 Provider Token 与 Session 管理；
- 首次登录只创建身份，不自动上传 localStorage 账本；
- 云端资金数据、首次迁移选择和多设备冲突仍属于下一阶段。

同一邮箱来自不同 Provider 时只允许 Better Auth 基于 Provider 返回的“邮箱已验证”状态执行安全关联；不信任未验证的邮箱声明。后续如开放手动关联，必须要求用户已有登录状态。

## 七、错误与可用性

- 用户拒绝授权：返回首页并显示统一提示；
- Provider 未配置：服务端拒绝对应登录，前端不展示按钮；
- OAuth 回调失败：不创建云端资金数据，不影响游客本地账本；
- Session 失效：回到游客状态，本地账本保持不变；
- GitHub 或 Google 临时不可用：其他 Provider 和游客模式继续可用。

## 八、实施顺序

### S1：移除邮箱流程

- 删除邮件、密码和 Turnstile 认证代码；
- 服务端关闭 `emailAndPassword`；
- 清理页面、测试和文档状态。

### S2：社交登录骨架

- 配置 Better Auth GitHub / Google Provider；
- 增加 Provider 能力接口；
- 登录抽屉改为 OAuth 按钮；
- 完成单元测试和本地 Pages 验收。

### S3：GitHub 生产配置

- 创建 GitHub OAuth App；
- 安全写入 Client ID 和 Secret；
- 验收授权、回调、Session、刷新和退出。

### S4：Google 生产配置

- 创建 Google Cloud OAuth Client；
- 配置 Consent Screen、正式域名和回调地址；
- 安全写入 Client ID 和 Secret；
- 验收授权与错误回退。

## 九、完成标准

- 页面和 API 不再提供邮箱密码注册；
- 未配置 Provider 不出现在登录界面；
- GitHub、Google 可独立启用或停用；
- OAuth 成功后刷新页面保持 Session；
- 退出不删除本地数据；
- 登录失败不影响游客模式；
- CI、Pages 自动部署和正式地址保持不变；
- 不新增服务器费用。

## 十、用户介入节点

S1、S2 不需要用户操作。进入 S3、S4 时需要用户在对应平台登录并确认创建 OAuth 应用。Client Secret 通过安全配置写入 Cloudflare，禁止粘贴到聊天中。
